# 🔄 시스템 동작 플로우

> **사용자의 지시에 따라 AI 에이전트 팀이 어떻게 동작하는지 상세히 설명합니다.**

---

## 전체 실행 흐름

```
사용자 아이디어 입력
        │
        ▼
┌──────────────────┐
│  🔧 시스템 초기화  │  스킬 등록 (9개) + 에이전트 로드 (7개)
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│  🎯 PM 분석       │  아이디어 분석 → 파이프라인 결정
│  (Orchestrator)   │  UI 키워드 감지 → 파이프라인 선택
└────────┬─────────┘
         │
         ├─── UI 키워드 없음 ──→ 기본 파이프라인
         │                      Planner → Architect → Coder → Reviewer → Tester
         │
         └─── UI 키워드 있음 ──→ UI 파이프라인
                                Planner → Architect → Designer → Coder → Reviewer → Tester
         │
         ▼
┌──────────────────┐
│  📝 Planner       │  아이디어 → PRD(기획서) 변환
│   (기획 단계)      │  5W1H 분석, 기능 목록, MVP 정의
└────────┬─────────┘
         │ 산출물: PRD
         ▼
┌──────────────────┐
│  🏗️ Architect     │  PRD → 설계서 변환
│   (설계 단계)      │  아키텍처, 데이터 모델, API, 폴더 구조
└────────┬─────────┘
         │ 산출물: 설계서
         ▼
┌──────────────────┐
│  🎨 Designer      │  (UI 프로젝트만) 
│   (UI 설계)       │  디자인 시스템, 컬러, 타이포, 컴포넌트
└────────┬─────────┘
         │ 산출물: UI 설계서
         ▼
┌──────────────────┐
│  💻 Coder         │  설계서 기반 코드 구현
│   (구현 단계)      │  PRD + 설계서 + UI 설계 참조
└────────┬─────────┘
         │ 산출물: 코드
         ▼
┌──────────────────┐     ❌ REJECTED
│  🔍 Reviewer      │────────────────┐
│   (리뷰 단계)      │                │
└────────┬─────────┘                │
         │ ✅ APPROVED              │
         ▼                         ▼
┌──────────────────┐     ┌─────────────────┐
│  🧪 Tester        │     │  💻 Coder       │
│   (테스트 단계)    │     │  (수정 재작업)   │
└────────┬─────────┘     └────────┬────────┘
         │                        │
         │ ✅ PASSED    (최대 3회) │
         │        ❌ FAILED ──────┘
         ▼
┌──────────────────┐
│  🎯 PM 보고       │  최종 보고서 생성
│   (완료 단계)      │  산출물 저장 + 메시지 히스토리 기록
└────────┬─────────┘
         │
         ▼
    📋 최종 보고서 출력
```

## 단계별 상세 설명

### 1단계: 시스템 초기화

```python
# main.py 실행 시 자동으로 수행됩니다.
python main.py
```

| 작업 | 내용 |
|------|------|
| 스킬 등록 | 9개 스킬(파일조작, 코드분석, 테스트, 문서생성)을 SkillsRegistry에 등록 |
| 에이전트 로드 | 7개 에이전트의 config.json + instructions.md를 읽어 AIAgent 인스턴스 생성 |
| 오케스트레이터 생성 | PM의 두뇌 역할을 하는 Orchestrator 초기화 |

---

### 2단계: PM 분석 (Orchestrator)

PM이 사용자 아이디어를 분석하여 프로젝트 유형을 판단합니다.

**UI 키워드 감지 목록:**
`웹`, `앱`, `UI`, `화면`, `페이지`, `프론트엔드`, `디자인`, `인터페이스`, `사이트`, `홈페이지`, `대시보드`

| 아이디어 | 감지 키워드 | 선택 파이프라인 |
|----------|------------|----------------|
| "할 일 관리 **웹앱**을 만들어줘" | `웹앱` ✅ | UI 파이프라인 (Designer 포함) |
| "주식 데이터를 분석하는 스크립트" | 없음 ❌ | 기본 파이프라인 |
| "**대시보드** 페이지 만들어줘" | `대시보드` ✅ | UI 파이프라인 (Designer 포함) |

---

### 3단계: Planner → PRD 생성

**입력**: 사용자 아이디어 원문
**출력**: PRD(기획서) 문서

```
컨텍스트 전달:
  PM → Planner: { "idea": "사용자 아이디어 원문" }
```

**PRD에 포함되는 내용:**
- 5W1H 분석 (누가, 무엇을, 왜, 언제, 어디서, 어떻게)
- 기능 목록 (P0 필수 / P1 중요 / P2 선택)
- MVP 범위 정의
- 사용자 시나리오 (정상 흐름 + 예외 흐름)
- 기술 제약사항

---

### 4단계: Architect → 설계서 생성

**입력**: 아이디어 + PRD
**출력**: 기술 설계서

```
컨텍스트 전달:
  PM → Architect: { "idea": "...", "prd": "PRD 전문" }
```

**설계서에 포함되는 내용:**
- 시스템 아키텍처 (레이어드 구조)
- 데이터 모델 (Python 클래스)
- API 설계 (엔드포인트, 메서드)
- 폴더 구조 (파일 트리)
- 기술 스택 + 선정 사유
- 설계 결정 (Trade-offs)

---

### 5단계: Designer → UI 설계 (선택적)

**UI 프로젝트에서만 실행됩니다.**

**입력**: 아이디어 + PRD + 설계서
**출력**: UI/UX 설계서

```
컨텍스트 전달:
  PM → Designer: { "idea": "...", "prd": "...", "architecture": "..." }
```

**UI 설계에 포함되는 내용:**
- 컬러 팔레트 (Primary, Secondary, BG, Surface, Text)
- 타이포그래피 (제목, 본문, 캡션)
- 레이아웃 (모바일 퍼스트, 12컬럼 그리드)
- 컴포넌트 목록
- 접근성 기준

---

### 6단계: Coder → 코드 구현

**입력**: 아이디어 + PRD + 설계서 + UI 설계
**출력**: Python 코드

```
컨텍스트 전달:
  PM → Coder: { "idea": "...", "prd": "...", "architecture": "...", "ui_design": "..." }
```

**Coder의 작업:**
- 설계서의 폴더 구조 생성
- 데이터 모델 구현
- 비즈니스 로직 작성
- SOLID 원칙 + 클린 코드 준수

---

### 7단계: Reviewer → 코드 리뷰

**입력**: PRD + 설계서 + 코드
**출력**: 리뷰 결과 (APPROVED / REJECTED)

```
컨텍스트 전달:
  PM → Reviewer: { "prd": "...", "architecture": "...", "code": "..." }
```

**리뷰 체크리스트:**
| 항목 | 검사 내용 |
|------|----------|
| 🔒 보안 | 하드코딩된 키, SQL 인젝션 |
| ⚡ 성능 | 불필요 반복, 메모리 누수 |
| 🛡️ 에러 | 예외 처리, 빈 except |
| 📐 설계 | 설계서 준수 여부 |
| ♻️ 중복 | 코드 중복 |

**분기:**
- `APPROVED` → Tester로 진행
- `REJECTED` → Coder에게 수정 피드백 전달 (최대 3회)

---

### 8단계: Tester → 테스트 실행

**입력**: PRD + 코드 + 리뷰 결과
**출력**: 테스트 결과 (PASSED / FAILED)

```
컨텍스트 전달:
  PM → Tester: { "prd": "...", "code": "...", "review": "..." }
```

**테스트 종류:**
1. 단위 테스트 (Unit) — 개별 함수 검증
2. 통합 테스트 (Integration) — 모듈 간 연동 검증
3. E2E 테스트 — 전체 흐름 검증 (해당 시)

**분기:**
- `ALL TESTS PASSED` → PM에게 완료 보고
- `TESTS FAILED` → Coder에게 버그 리포트 전달 (최대 3회)

---

### 9단계: 최종 보고서

PM이 전체 결과를 취합하여 최종 보고서를 생성합니다.

**보고서 내용:**
- 프로젝트명
- 원본 아이디어
- 사용된 파이프라인
- 피드백 루프 횟수
- 각 산출물 현황 (✅ 완료 / ⬜ 생략)

**파일 저장:**
```
shared/context/
├── {프로젝트명}_context.json    ← 전체 컨텍스트 (산출물 포함)
└── {프로젝트명}_messages.json   ← 메시지 히스토리
```

## 피드백 루프 상세

Reviewer나 Tester가 문제를 발견하면 **자동으로 Coder에게 수정을 요청**합니다.

```
[피드백 루프]

Coder (1차 구현)
    │
    ▼
Reviewer ──── REJECTED ───→ Coder (2차 수정)
    │                           │
    ▼ APPROVED                  ▼
Tester                     Reviewer ──── REJECTED ───→ Coder (3차 수정)
    │                                                     │
    ▼ PASSED                                              ▼
PM (완료)                                          ⚠️ 최대 3회 도달
                                                   현재 상태로 진행
```

**핵심 규칙:**
- 최대 반복: **3회** (config.py에서 `MAX_FEEDBACK_LOOPS`로 조절 가능)
- 3회 도달 시: 경고를 출력하고 현재 상태로 다음 단계 진행

## 컨텍스트 전달 체인

에이전트 간 전달되는 컨텍스트의 흐름:

```
사용자 아이디어
    │
    ├───→ Planner:   { idea }
    │         │
    │         └── PRD 생성
    │
    ├───→ Architect: { idea, prd }
    │         │
    │         └── 설계서 생성
    │
    ├───→ Designer:  { idea, prd, architecture }
    │         │
    │         └── UI 설계서 생성
    │
    ├───→ Coder:     { idea, prd, architecture, ui_design }
    │         │
    │         └── 코드 생성
    │
    ├───→ Reviewer:  { prd, architecture, code }
    │         │
    │         └── 리뷰 결과
    │
    └───→ Tester:    { prd, code, review }
              │
              └── 테스트 결과
```

> **핵심**: 각 에이전트는 자신의 작업에 필요한 **이전 단계 산출물만** 전달받습니다.
> Planner는 아이디어만, Coder는 모든 이전 산출물을 참조합니다.

## 메시지 버스

모든 에이전트 간 통신은 **메시지 버스**를 통해 기록됩니다.

```python
Message(
    sender="planner",           # 보낸 에이전트
    receiver="architect",       # 받는 에이전트
    content="PRD 문서 내용...",  # 메시지 본문
    msg_type="result",          # 유형: request/result/feedback/error
    context={"idea": "..."},    # 누적 컨텍스트
    timestamp="2026-04-16..."   # 시각
)
```

**메시지 유형:**
| 유형 | 의미 | 예시 |
|------|------|------|
| `request` | 작업 요청 | PM → Planner: "기획서 작성해줘" |
| `result` | 작업 결과 | Planner → Architect: "PRD 완성" |
| `feedback` | 수정 요청 | Reviewer → Coder: "보안 이슈 수정해줘" |
| `error` | 오류 발생 | Agent → PM: "스킬 실행 실패" |
